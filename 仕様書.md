## 目的

StreetVoice を Google TV で使うための **個人利用前提の PoC** を作りたい。
まずは UI を作り込まず、**検索 → 曲選択 → ストリーム URL 取得 → 再生** の最短経路を検証する。

StreetVoice には Android アプリはあるが、Google Play 上では少なくとも phone / tablet 向け表示で、TV 対応が前面に出ていない。なので、TV での利用を補う薄いクライアントを試作したい。 ([Google Play][2])

---

## 前提条件

* これは製品化前提ではなく **技術検証用 PoC**
* 公式公開 API はない前提で進める
* StreetVoice の Web クライアントが内部的に使っている通信を観察し、その再利用可能性を検証する
* 壊れやすいことを前提に、**実装は薄く・疎結合にする**
* ログイン機能やライブラリ同期は後回し
* まずは **未ログイン状態で 1 曲再生できるか** がゴール

---

## ゴール

### 最低ゴール

以下が通れば成功とする。

1. StreetVoice 上の曲を検索できる
2. 検索結果から曲を 1 件選べる
3. 曲詳細を取得できる
4. 再生用 URL を取得できる
5. Google TV 向け Android アプリ上で再生できる

### 成功条件

* 任意のキーワードで検索できる
* 取得した song id から曲詳細 JSON を取れる
* HLS などの再生 URL を取得できる
* ExoPlayer / Media3 で 1 曲再生できる
* 10 曲程度の簡易リストで次の曲に進める

---

## 非ゴール

今回やらないこと。

* 正式な公開 API クライアントを名乗ること
* 完全な StreetVoice 互換アプリを作ること
* アカウント連携、お気に入り同期、コメント、通知
* 商用利用、公開配布前提の品質作り
* 先に綺麗な TV UI を作り込むこと

---

## 技術方針

### 実装を 2 段階に分ける

#### Phase 1: 調査・検証スクリプト

まず TypeScript / Node.js で以下を確認する。

* 検索リクエストの再現
* 曲詳細 API の再現
* 再生 URL 取得 API の再現
* Cookie / Referer / CSRF 依存の有無
* URL の寿命

#### Phase 2: 最小 Android TV クライアント

Kotlin + Compose for TV + Media3 で以下だけ作る。

* 検索画面
* 検索結果一覧
* 曲再生
* 再生 / 一時停止 / 次へ

### スタック

* 調査スクリプト: TypeScript, Node.js, fetch, cheerio
* TV クライアント: Kotlin, Compose for TV, Media3 ExoPlayer
* API 呼び出し部分は repository / datasource に隔離
* エンドポイントやレスポンスは将来変わる前提なので直書きを最小限にする

### 2. 調査スクリプトを先に作る

#### 要件

* キーワード検索の入力を受ける
* StreetVoice Web の検索結果を取得する
* 検索結果 HTML または API レスポンスから

  * title
  * artist
  * song id
  * song url
    を抽出する
* song id から曲詳細取得を試す
* song id から再生 URL 取得を試す
* レスポンス JSON / HTML を `debug/` に保存する
* 失敗時には HTTP status, headers, body の要約を出す

#### 実装方針

* 最初は `fetch` ベース
* 必要なら `undici` でもよい
* 解析は `cheerio`
* request header は最小限から始める

  * user-agent
  * referer
  * accept
  * x-requested-with
* いきなり複雑な抽象化は不要
* まず動く調査コードを優先

---

### 3. 調査スクリプトで確認してほしい観点

#### A. 検索

* Web 検索は HTML 返却か JSON 返却か
* song id は DOM 上にあるか
* song ページ URL から song id を抜けるか
* ページネーションがあるか

#### B. 曲詳細

* 未ログインで叩けるか
* Referer 必須か
* rate limit の気配があるか
* レスポンス JSON の shape を型定義化できるか

#### C. 再生 URL

* 未ログインで取れるか
* POST が必要か
* HLS の m3u8 が返るか
* 署名付き URL か
* 数分後に失効するか
* TV アプリ側でそのまま再生できるか

#### D. 認証依存

* Cookie なしでどこまで行けるか
* CSRF token が必要か
* ブラウザセッション必須か

## 期待する設計

### repository の境界を切る

StreetVoice の不安定な内部 API を直接 UI から呼ばないでほしい。
例えばこんな感じ。

```ts
interface StreetVoiceRepository {
  searchSongs(query: string): Promise<SongSummary[]>;
  getSongDetail(songId: string): Promise<SongDetail>;
  getPlayableStream(songId: string): Promise<PlayableStream>;
}
```

Android 側も同様に、実際の取得ロジックは datasource に閉じ込める。

---

## 失敗時の扱い

Claude Code には、失敗したら無理に実装を進めず、**どこで詰まったかを明示**してほしい。

最低限このどれかに分類すること。

* 検索は成功、再生 URL 取得が失敗
* 曲詳細までは成功、HLS 取得が認証依存
* URL は取得できたが Media3 再生不可
* CORS / header / cookie 条件が厳しい
* WebView セッション経由なら可能そう

そして、各失敗ごとに次の一手を提案してほしい。

---

## 注意事項

* ハードコードしすぎないこと
* 実通信を見て確かめるまで API shape を決め打ちしないこと
* 「外部実装にあったから」で確定扱いしないこと
* debug ログと保存データを必ず残すこと
* 実装を綺麗にしすぎるより、まず観測可能性を優先すること
* 変更に弱い内部 API なので、可観測性と分離を優先すること

---

## 参考仮説

以下は確定事項ではなく、**最初に当たる仮説**として扱ってほしい。

* `song detail` 相当の API がある可能性が高い
* `song の HLS file` 相当の API がある可能性が高い
* 再生 URL は短命の可能性がある
* `Referer` や Cookie が必要な可能性がある
* 未ログインで 1 曲再生までできるなら PoC として十分

この仮説は、外部実装で StreetVoice クライアントが存在していること、および StreetVoice 側で `/api/` が内部向け領域として扱われていることから置いている。 ([GitHub][3])

## まず最初にやるタスク

Claude Code は以下の順で進めてほしい。

1. `research/` の TypeScript スクリプトを作成
2. 検索・曲詳細・再生 URL 取得の調査コードを書く
3. debug 出力を保存する
4. `docs/findings.md` を更新
5. 成功した最小フローが見えたら Android TV アプリの skeleton を作る
6. 1 曲再生まで繋ぐ
7. 最後に、現時点でのリスクと継続可否を `docs/risks.md` にまとめる

---

## 出力してほしいもの

最終的に Claude Code には以下を返してほしい。

* 作成したファイル一覧
* 調査結果サマリ
* 実際に使えたエンドポイント候補
* 必須ヘッダの有無
* Cookie / 認証の要否
* 1 曲再生まで成功したか
* 次にやるべきこと
* 継続価値があるかどうかの判断
